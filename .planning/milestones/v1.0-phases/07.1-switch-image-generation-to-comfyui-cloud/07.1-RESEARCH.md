## Summary

The ComfyUI Cloud provider skeleton is already implemented and substantially correct, but has one **critical functional bug** that will cause all image generation attempts to silently fail, plus several smaller correctness issues. The critical bug is in `_poll_and_download`: when a job completes, the code tries to extract `outputs` from the status response body, but the cloud.comfy.org `/api/job/{id}/status` endpoint only returns `{"status": "..."}` with no image data. Image outputs must be fetched via a separate `GET /api/history_v2/{prompt_id}` call once status is `completed`. This is confirmed by the official ComfyUI Cloud API docs and Python examples.

Secondary issues include: missing seed randomization (causing identical images on repeated calls), a Protocol signature mismatch (`ImageProvider` Protocol omits `reference_image_url` but the processor passes it), stale references to Replicate in `processor.py` docstring and `.env.example`, and the `replicate` package still listed in `requirements.txt`. The workflow JSON injection points and API authentication (X-API-Key header) are verified correct.

**Primary recommendation:** Fix `_poll_and_download` to call `/api/history_v2/{prompt_id}` for outputs, add seed randomization, update the Protocol, clean up Replicate artifacts, and write a unit-testable smoke test with mocked HTTP.

### Pattern 1: Correct ComfyUI Cloud API Flow (4-step, not 3-step)

The current code implements 3 steps (submit → poll → download-from-status). The correct flow is 4 steps:

```
Step 1: POST /api/prompt             → gets prompt_id
Step 2: GET /api/job/{id}/status     → poll until "completed"  (returns only status string)
Step 3: GET /api/history_v2/{id}     → fetch outputs JSON with node IDs and filenames
Step 4: GET /api/view?filename=...   → download image bytes (follows 302 redirect)
```

**What:** The status endpoint confirms completion. The history endpoint provides the output file metadata. These are two separate API calls.

**Example (official Python from docs.comfy.org):**
```python
# Source: https://docs.comfy.org/development/cloud/overview

# Poll until complete (status endpoint returns {"status": "..."} only)
while True:
    status = requests.get(f"{BASE_URL}/api/job/{prompt_id}/status",
                          headers={"X-API-Key": API_KEY}).json()
    if status["status"] == "completed":
        break

# Fetch outputs separately
history = requests.get(f"{BASE_URL}/api/history_v2/{prompt_id}",
                       headers={"X-API-Key": API_KEY}).json()
outputs = history["outputs"]

# Download by node ID
images = outputs["9"]["images"]   # node "9" is SaveImage
file_info = images[0]
# GET /api/view?filename=...&subfolder=...&type=output  (302 redirect to signed URL)
```

### Pattern 3: Seed Randomization

Seeds are large integers in the workflow JSON. Python's `random.randint` or `secrets.randbelow` works:

```python
import random

def _build_t2i(self, prompt: str) -> tuple[dict, str]:
    workflow = copy.deepcopy(_load_workflow("text_to_image"))
    workflow["91"]["inputs"]["value"] = prompt
    workflow["86:3"]["inputs"]["seed"] = random.randint(0, 2**32 - 1)
    return workflow, "60"

def _build_i2i(self, ...) -> tuple[dict, str]:
    workflow = copy.deepcopy(_load_workflow("image_to_image"))
    workflow["41"]["inputs"]["image"] = ref_filename
    workflow["89:68"]["inputs"]["prompt"] = prompt
    workflow["89:65"]["inputs"]["seed"] = random.randint(0, 2**32 - 1)
    return workflow, "9"
```

`random.randint` is sufficient — no cryptographic strength needed for image seeds.

### Pattern 4: ImageProvider Protocol Fix

The current `ImageProvider` Protocol in `base.py` defines:
```python
async def generate(self, prompt: str, aspect_ratio: str = "2:3") -> GeneratedImage: ...
```

But `ComfyUIProvider.generate()` and `processor.py` both pass `reference_image_url=...`. Python's structural typing (Protocol) checks structural compatibility. With `runtime_checkable`, `isinstance(provider, ImageProvider)` checks the method name exists — it does NOT enforce parameter signatures at runtime. So the Protocol check does NOT fail, but the Protocol is misleading/incorrect as documentation.

**Recommended fix:** Update the Protocol to include `reference_image_url: str | None = None` so it accurately describes the extended interface. `ReplicateProvider` doesn't support it, but since it's optional with a default of `None`, `ReplicateProvider.generate()` remains structurally compatible (extra kwargs are ignored in Python duck typing for Protocol checks).

### Pattern 5: Auth and Upload Response

All confirmed from official docs:
- **Auth header:** `X-API-Key: <key>` on every request — CORRECT in current code
- **Submit:** `POST /api/prompt` with `{"prompt": workflow}` — CORRECT, returns `{"prompt_id": "..."}`
- **Upload response:** `{"name": "filename.jpg", "subfolder": "..."}` — CORRECT, code reads `upload_resp.json()["name"]`
- **View redirect:** `GET /api/view?...` returns 302 → code uses `follow_redirects=True` — CORRECT

### Anti-Patterns to Avoid
- **Reading outputs from status body:** The current `_poll_and_download` passes `data` (the status JSON) to `_download_output`, but status body is `{"status": "completed"}` only. This is the critical bug.
- **Hardcoded seeds:** Fixed seeds produce identical images every time. Must randomize per generation.
- **Removing the fallback URL path in processor.py:** The `if generated.image_bytes: ... else: download from URL` logic in processor.py is valid — keep it for forward compatibility.

## Common Pitfalls

### Pitfall 1: Status-Only Response (CRITICAL — is already broken)
**What goes wrong:** `_poll_and_download` reads `data.get("outputs", {})` from the status response. The status endpoint returns `{"status": "completed"}` only — no `outputs` key. The `images = []` branch triggers, raising `RuntimeError: ComfyUI: no images found in output node '60'`.
**Why it happens:** The code was written assuming status embeds outputs (like Replicate does). Cloud ComfyUI separates status from history.
**How to avoid:** After status == "completed", call `GET /api/history_v2/{prompt_id}` and read `history["outputs"]`.
**Warning signs:** Logs show `No images in output node 60. Full status: {"status": "completed"}`.

### Pitfall 2: Output Persistence Race Condition
**What goes wrong:** The `/api/history_v2/{prompt_id}` response immediately after status=completed may return empty `outputs` because cloud storage write hasn't finished.
**Why it happens:** Documented in GitHub issue #11540 on ComfyUI repo — "execution_success fires before outputs are persisted."
**How to avoid:** If `history["outputs"]` is empty on first fetch after completed, add a short retry (1-2 seconds, 2-3 attempts) before raising RuntimeError.
**Warning signs:** `outputs: {}` on history call immediately after completed status.

### Pitfall 3: Hardcoded Seeds = Identical Images
**What goes wrong:** Every call to `_build_t2i` or `_build_i2i` injects the same hardcoded seed from the JSON file. Every generation call produces the same output image.
**Why it happens:** The workflow JSONs have fixed seeds that were not overridden at runtime.
**How to avoid:** Always override `workflow["86:3"]["inputs"]["seed"]` and `workflow["89:65"]["inputs"]["seed"]` with fresh `random.randint()` on each build.

### Pitfall 4: Protocol Mismatch Masking Bugs
**What goes wrong:** `isinstance(provider, ImageProvider)` returns True even though `ImageProvider` Protocol doesn't declare `reference_image_url`. This hides the mismatch during development.
**Why it happens:** Python runtime Protocol checks only verify method names, not signatures.
**How to avoid:** Update the Protocol to match the actual interface used by callers. ReplicateProvider stays compatible (its `generate` has fewer parameters — that's fine for Protocol structural typing).

### Pitfall 5: `.env.example` Documents Replicate as Required
**What goes wrong:** `.env.example` says "--- Replicate (required for image generation) ---" with `REPLICATE_API_TOKEN=r8_...`. New developers will try to get a Replicate key when they should be getting a ComfyUI Cloud key.
**How to avoid:** Update `.env.example` to replace Replicate section with ComfyUI Cloud section (`COMFYUI_API_KEY`, `COMFYUI_BASE_URL`).

### Pitfall 6: `replicate` Package Import Errors
**What goes wrong:** `replicate` in `requirements.txt` has known Pydantic v2/Python 3.14 incompatibility (already documented in `replicate_provider.py` module docstring). Even as an unused dep, it may cause import-time warnings or installation failures.
**How to avoid:** Remove `replicate` from `requirements.txt` if ReplicateProvider is no longer the active provider. The lazy import trick in `replicate_provider.py` defers failures to call time — but if no code calls it, better to just remove the dep.

### Pitfall 7: Stale Docstrings Mislead Future Readers
**What goes wrong:** `processor.py` module docstring says "Call Replicate FLUX API" and "Step 2: Call Replicate FLUX API". A developer reading this will be confused about which provider is active.
**How to avoid:** Update the `processor.py` module docstring to accurately describe the ComfyUI Cloud flow.

## Code Examples

### Corrected `_poll_and_download` Pattern
```python
# Source: https://docs.comfy.org/development/cloud/overview (official Python example)

async def _poll_and_download(
    self, client: httpx.AsyncClient, prompt_id: str, output_node: str
) -> bytes:
    """Poll until completed, fetch history, then download output image bytes."""
    elapsed = 0.0
    while elapsed < POLL_TIMEOUT:
        await asyncio.sleep(POLL_INTERVAL)
        elapsed += POLL_INTERVAL

        resp = await client.get(
            f"{settings.comfyui_base_url}/api/job/{prompt_id}/status",
            headers=self._headers(),
        )
        resp.raise_for_status()
        data = resp.json()
        status = data.get("status")
        logger.debug(f"ComfyUI job {prompt_id} status: {status}")

        if status == "completed":
            # Status body is {"status": "completed"} only.
            # Fetch outputs from history endpoint.
            return await self._fetch_history_and_download(client, prompt_id, output_node)
        if status in ("failed", "cancelled", "canceled"):
            raise RuntimeError(
                f"ComfyUI job {status}: {data.get('error') or data}"
            )

    raise TimeoutError(f"ComfyUI job timed out after {POLL_TIMEOUT}s")


async def _fetch_history_and_download(
    self, client: httpx.AsyncClient, prompt_id: str, output_node: str
) -> bytes:
    """Fetch /api/history_v2/{prompt_id} and download the image from output node."""
    # Retry loop to handle race condition where outputs may not yet be persisted
    for attempt in range(3):
        history_resp = await client.get(
            f"{settings.comfyui_base_url}/api/history_v2/{prompt_id}",
            headers=self._headers(),
        )
        history_resp.raise_for_status()
        history = history_resp.json()
        outputs = history.get("outputs", {})
        if outputs:
            break
        logger.warning(f"ComfyUI history outputs empty (attempt {attempt + 1}/3), retrying...")
        await asyncio.sleep(2.0)

    node_out = outputs.get(output_node, {})
    images = node_out.get("images", [])
    if not images:
        logger.error(f"No images in output node {output_node}. Full history: {history}")
        raise RuntimeError(
            f"ComfyUI: no images found in output node '{output_node}'. "
            f"Keys available: {list(outputs.keys())}"
        )

    img_meta = images[0]
    filename = img_meta["filename"]
    subfolder = img_meta.get("subfolder", "")

    view_resp = await client.get(
        f"{settings.comfyui_base_url}/api/view",
        params={"filename": filename, "subfolder": subfolder, "type": "output"},
        headers=self._headers(),
        follow_redirects=True,
    )
    view_resp.raise_for_status()
    logger.info(f"Downloaded {len(view_resp.content)} bytes from ComfyUI output")
    return view_resp.content
```

## Open Questions

1. **Keep or remove ReplicateProvider?**
   - What we know: No active code path uses it. `replicate` package has known Pydantic v2/Python 3.14 issues. CONTEXT.md marks this as "Claude's Discretion."
   - Recommendation: Keep `replicate_provider.py` file (cheap safety net) but remove the `replicate` package from `requirements.txt` since no code currently imports it. The lazy import in `replicate_provider.py` means it won't break at startup. If Replicate is needed as emergency fallback, restore the dep then.

2. **History endpoint race condition severity?**
   - What we know: GitHub issue #11540 documents that `execution_success` can fire before outputs are persisted. The retry loop (3 attempts × 2s) adds at most 6s extra latency when it occurs.
   - Recommendation: Implement the 3-attempt retry with 2s sleep. This is defensive and low-cost.

3. **Smoke test structure?**
   - What we know: CONTEXT.md says "Claude's Discretion." The project uses pytest with `pytest-asyncio`. No live API key will be available in CI.
   - Recommendation: Unit test with `unittest.mock.AsyncMock` patching `httpx.AsyncClient`. Test the full `generate()` path including the new `_fetch_history_and_download` method. One test for t2i, one for i2i. Do not require a live `COMFYUI_API_KEY`.

4. **Does `/api/history_v2` always return `outputs` as top-level key?**
   - What we know: Official Python example shows `history["outputs"]`. The response schema shows `{"outputs": {...}}`.
   - Confidence: HIGH based on official docs. But real-world API calls may reveal edge cases (e.g., failed job history structure).
