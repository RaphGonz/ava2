## Accomplishments

- Critical bug fixed: `_poll_and_download` now calls `/api/history_v2/{prompt_id}` after `status=completed` instead of reading outputs from the status body (which only ever contains `{"status": "..."}`)
- Added `_fetch_history_and_download` with 3-attempt retry loop for the output persistence race condition documented in ComfyUI GitHub issue #11540
- Added `random.randint(0, 2**32 - 1)` seed injection in both `_build_t2i` (node `86:3`) and `_build_i2i` (node `89:65`) — repeated calls now produce different images
- Updated `ImageProvider` Protocol to include `reference_image_url: str | None = None` — accurately documents the interface used by both ComfyUIProvider and processor.py callers
- 5-test unit suite covers: t2i end-to-end, i2i end-to-end, seed randomization, Protocol compliance, history retry on empty outputs — all with mocked httpx, no live COMFYUI_API_KEY needed
- Full 52-test backend suite passes with no regressions

## Decisions Made

- Kept `_download_output` as a private helper method called by `_fetch_history_and_download` rather than inlining the view download logic — avoids code duplication and keeps methods single-responsibility
- Used `POLL_INTERVAL` (4s) for the history retry sleep rather than a separate constant — consistent with the rest of the polling code; 3 retries × 4s = 12s max additional latency for race condition
- `random.randint` (not `secrets.randbelow`) for seed generation — image seeds are not a security use case; `random` is simpler and sufficient
