# Phase 7.1: Switch image generation to ComfyUI Cloud - Context

**Gathered:** 2026-02-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace Replicate FLUX as the image generation backend with ComfyUI Cloud (cloud.comfy.org) using Qwen models. The ComfyUIProvider skeleton, workflow JSON files, and processor wiring are already implemented. This phase verifies API compatibility, fixes remaining issues (seed randomization, any schema mismatches), validates end-to-end pipeline correctness, and removes all Replicate dependencies from the active code path.

</domain>

<decisions>
## Implementation Decisions

### Model selection
- Text-to-image: `qwen_image_2512_fp8_e4m3fn.safetensors` (workflow: text_to_image.json, output node `60`)
- Image-to-image: `qwen_image_edit_2511_bf16.safetensors` (workflow: image_to_image.json, output node `9`)
- Both workflows already exist in `backend/app/services/image/workflows/`

### Workflow JSON injection points
- t2i prompt injection: `workflow["91"]["inputs"]["value"] = prompt` (node 91 is PrimitiveStringMultiline)
- i2i reference image: `workflow["41"]["inputs"]["image"] = uploaded_filename` (node 41 is LoadImage)
- i2i prompt injection: `workflow["89:68"]["inputs"]["prompt"] = prompt` (node 89:68 is TextEncodeQwenImageEditPlus)
- These injection points are already coded in `ComfyUIProvider._build_t2i` and `_build_i2i`

### Seed handling
- Randomize seed per generation (override hardcoded seeds in workflow JSONs at runtime)
- t2i seed is in `workflow["86:3"]["inputs"]["seed"]`
- i2i seed is in `workflow["89:65"]["inputs"]["seed"]`

### Prompt construction
- Keep existing FLUX-style `build_avatar_prompt` — no change to prompt_builder.py needed

### API compatibility
- cloud.comfy.org API endpoints are untested — needs integration test against live API
- Current code assumes: POST `/api/prompt`, GET `/api/job/{id}/status`, GET `/api/view`, POST `/api/upload/image`
- If endpoints differ, `ComfyUIProvider` methods `_submit`, `_poll_and_download`, `_upload_image`, `_download_output` need updating

### Claude's Discretion
- How to structure the integration test (smoke test vs full mock)
- Whether to keep ReplicateProvider as a fallback or remove it entirely
- How to handle the ImageProvider Protocol signature mismatch (`reference_image_url` not in Protocol but used in processor)

</decisions>
